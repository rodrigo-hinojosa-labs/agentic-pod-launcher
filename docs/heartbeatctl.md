# heartbeatctl — reference

CLI for inspecting, controlling, and mutating the in-container heartbeat. Image-baked at `/opt/agent-admin/scripts/heartbeatctl`, symlinked to `/usr/local/bin/heartbeatctl`.

## Invocation

**Always pass `-u agent` to docker exec.** The container runs with `cap_drop: ALL` + `no-new-privileges: true`, and several heartbeatctl paths (`reload`, `pause`, `set-*`) write to agent-owned files. The default `docker exec` is root, which cannot write those without `CAP_DAC_OVERRIDE` / `CAP_FOWNER`.

```bash
docker exec -u agent <agent-name> heartbeatctl <subcommand> [args]
# Inside the tmux session it's on PATH:
heartbeatctl <subcommand> [args]
```

## Read commands

### `status [--json]`

Dashboard. Reads `state.json` and enriches with live `crond.alive` / `pid` at read time.

Pretty output (no flag):

```
heartbeat: ENABLED
  interval: 2m   cron: */2 * * * *
  prompt:   Check status and report to notification telegram bot
  notifier: telegram
  last run: ok  @ 2026-04-19T19:36:00Z  (125000ms, attempt 1)
  counters: runs=2 ok=2 timeout=0 error=0 streak_fail=0  24h=100%
  crond:    alive (pid 16)
  updated:  2026-04-19T19:38:05Z

recent runs:
  2026-04-19T19:19:14Z  ok  73000ms   Check status and report...
  2026-04-19T19:36:00Z  ok  125000ms  Check status and report...
```

`--json`: emits `state.json` verbatim for machine parsing (counters, last_run object, crond liveness).

Exit codes:

- `0` on success.
- `1` if `state.json` schema is unknown (refuses to guess at incompatible formats).
- `2` if `state.json` is missing (heartbeat has never run — try `heartbeatctl reload`).

### `logs [-n N] [--json]`

Tail `logs/runs.jsonl`. Default `N=20`.

Pretty output (no flag): a table of `TS | STATUS | DUR | ATTMPT | PROMPT[:60]`.

`--json`: emits raw JSON lines, one per run, piping-safe for `jq`.

### `show`

Dumps the three primary sources of truth so you can tell what's actually installed vs. what's configured:

- `heartbeat.conf` (shell-sourced by `heartbeat.sh`, regenerated by `reload`)
- `/etc/crontabs/agent` (what crond actually reads — root-owned)
- `agent.yml` slice: `features.heartbeat` + `notifications`

Useful for diagnosing "why is the schedule different from what I set".

## Control commands

### `test [--prompt "..."]`

Run one heartbeat right now. Writes a line to `runs.jsonl` with `trigger: manual`. Does **not** bump `counters.consecutive_failures` (a manual test must not poison the ok-streak).

Accepts `--prompt "override"` to test a different prompt without changing `agent.yml`.

Returns when the run completes — expect the typical duration (same as a cron tick).

### `pause`

Sets `features.heartbeat.enabled = false` in `agent.yml`, prefixes the crontab line with `# paused:`, and drops the cron dispatch. Idempotent — running it twice does not double-comment.

### `resume`

Inverse of `pause`. Sets `enabled = true`, uncomments the crontab line.

### `reload`

Re-reads `agent.yml`, regenerates `heartbeat.conf`, rewrites the crontab staging file, and ensures `logs/` exists. Does **not** signal crond (busybox crond dies on SIGHUP). crond picks up the new crontab on its own minute tick — propagation is described in the "Propagation timing" section below.

Idempotent. Called automatically by:

- `entrypoint.sh` on every container start (via `start_services.sh`).
- Every `pause` / `resume` / `set-*` subcommand after mutating `agent.yml`.

## Mutation commands

All mutations are transactional with automatic rollback on failure:

1. Validate the value. Reject with exit code 1 if invalid; `agent.yml` untouched.
2. Back up `agent.yml` to `agent.yml.prev`.
3. Apply the mutation via `yq -i`.
4. Run `reload`.
5. On any failure in steps 3–4, restore `agent.yml.prev` and re-run `reload` so the live state matches the backed-up config. Exit code 2.
6. On success, delete the backup.

### `set-interval <Nm|Nh>`

Accepts a restricted divisor table so the derived cron expression is always valid:

- `Nm` for `N` in `{1, 2, 3, 4, 5, 6, 10, 12, 15, 20, 30}` → `*/N * * * *` (or `* * * * *` for `1m`).
- `Nh` for `N` in `{1, 2, 3, 4, 6, 8, 12}` → `0 */N * * *` (or `0 * * * *` for `1h`).
- `24h` → `0 0 * * *` (once per day at 00:00 UTC).

Anything else is rejected. Examples that do NOT work: `30s`, `45m`, `7m`, `5h`, `2h30m`.

### `set-prompt "<str>"`

Updates `features.heartbeat.default_prompt`. Accepts the prompt as an arg or via stdin if no arg is given. Length 1..4000 chars.

Warns to stderr (but does not refuse) if the current `notifications.channel` is `telegram` and `.env` is missing `NOTIFY_BOT_TOKEN` / `NOTIFY_CHAT_ID`.

#### Prompt examples by use case

The default prompt (`Status check — return a short plain-text report (uptime, notable issues). No tool use; your stdout is forwarded verbatim to the notifier.`) is intentionally generic. Replace it via `heartbeatctl set-prompt "..."` to match what your agent actually does:

- **Cluster health** — *"List Kubernetes pods in CrashLoopBackOff or Error in the last 30m via `kubectl get pods -A`. Return only the failing pods with their namespace and reason. If everything is healthy, output `OK: N pods running`."*
- **Vault ingestion monitor** — *"Count `*.md` files under `~/.vault/wiki/` and compare against the last value in `~/.vault/log.md`. Report any drop or > 10% gain. Note any sync-conflict files in `~/.vault/`."*
- **Jira sprint blockers** — *"Use `mcp__atlassian__jira_search` to query the active sprint of project KEY. List blockers (status=Blocked or label=blocker). Group by assignee. Output one bullet per blocker with `[KEY-123] @user — title`."*
- **Log tailer** — *"Tail the last 50 lines of `/workspace/claude.log` and report any ERROR or WARN. Include timestamps and a one-line summary of patterns (e.g. `3× 'connection refused' between 14:05 and 14:18`)."*
- **Heartbeat-as-canary** — *"Reply with `pong @ <ISO timestamp>`. No tool use. This proves Claude Code is reachable and the notifier round-trip works."*

Each prompt should:
1. Be self-contained — heartbeat sessions run in an isolated `CLAUDE_CONFIG_DIR=/home/agent/.claude-heartbeat` with no plugins by default and no conversation history.
2. Constrain output length — heartbeat trims at 3500 chars before forwarding to the notifier; prompts that ramble waste the budget.
3. Be explicit about tools — say *"no tool use"* when you want a fast pulse, or *"use `mcp__X__Y`"* when you want a query.

### `set-notifier <none|log|telegram>`

Updates `notifications.channel`. Membership check — anything else rejected.

- `none` — silent; no-op notifier.
- `log` — appends to `scripts/heartbeat/logs/notifications.log`.
- `telegram` — sends via `NOTIFY_BOT_TOKEN` / `NOTIFY_CHAT_ID` (see note in `set-prompt`).

### `set-timeout <seconds>`

Integer ≥ 10. Max timeout the heartbeat waits for `claude --print` before killing the tmux session and recording `status: timeout`.

### `set-retries <N>`

Integer 0..5. Number of retries after a failed attempt. Total attempts = `retries + 1`.

## Propagation timing

When you mutate a field, the live schedule updates in two steps:

1. `heartbeatctl reload` writes the new crontab to the workspace staging file (`scripts/heartbeat/.crontab.staging`). Instantaneous.
2. The root-privileged sync loop in `entrypoint.sh` notices the staging file is newer than `/etc/crontabs/agent` and copies it there. Runs every 15 seconds.
3. busybox crond re-scans crontab mtimes on its minute tick and picks up the new schedule. Up to 60 seconds.

**Worst case total propagation: ~75 seconds.**

The staging/sync indirection exists because busybox crond silently skips any file in `/etc/crontabs/` that isn't root-owned, so agent cannot write there directly. Entrypoint runs as root before dropping privileges, so it's the only context that can install files into `/etc/crontabs/`.

## Data files

All under `/workspace/scripts/heartbeat/`:

- `heartbeat.conf` — shell-sourced by `heartbeat.sh`. Regenerated by `reload`.
- `state.json` — atomic snapshot of the last run + aggregate counters. Schema `1`.
- `.crontab.staging` — written by `reload`, consumed by entrypoint sync loop.
- `logs/runs.jsonl` — append-only JSON-per-line trace. Rotated at 10MB → `.1`, `.2.gz`, `.3.gz` (max 3 generations).
- `logs/cron.log` — crond stderr for schedule-dispatch debugging.
- `logs/sessions/*.log` — per-run tmux session output (last 20 retained).

## Backup commands

The agent's persistent state is sliced into three orphan branches on the
agent's own fork (`backup/identity`, `backup/vault`, `backup/config`).
Each primitive is idempotent, hash-based, and pushes nothing when its
input hasn't changed since the last commit. Restore happens via
`setup.sh --restore-from-fork <url>`, which pulls all three in order.

| Branch | Cadence | What it stores |
|---|---|---|
| `backup/identity` | watchdog 60s + post-plugin-install + cron 03:30 | OAuth login, Telegram pairing, plugin config, settings, age-encrypted `.env` |
| `backup/vault` | cron `0 * * * *` (override via `vault.backup_schedule`) | Markdown subset of the vault. Excludes `.obsidian/workspace*.json`, cache, `.trash/`, `*.sync-conflict-*` |
| `backup/config` | cron 03:30 (toggle via `features.config_backup.enabled`) | `agent.yml` (plaintext, no secrets) |

> **Freshness check**: `agentctl doctor` verifies that each backup pushed within its expected cadence × 2 (identity 48h, vault 25h, config 8d) and prints a humanized delta (`pushed 12h ago`). Override the thresholds for one-off runs via `DOCTOR_IDENTITY_MAX_AGE_HOURS`, `DOCTOR_VAULT_MAX_AGE_HOURS`, `DOCTOR_CONFIG_MAX_AGE_DAYS`. `heartbeatctl status` prints the same `last_push` timestamps for all three branches under their respective sections.

### `backup-identity`

Snapshot the agent's identity subset (login, pairing, plugin config,
settings, optionally age-encrypted `.env`) to the `backup/identity`
orphan branch on the fork.

```
heartbeatctl backup-identity                       # default: run
heartbeatctl backup-identity --configure-key KEY   # set recipient + backup
heartbeatctl backup-identity --disable             # stop scheduled backups
heartbeatctl backup-identity --dry-run             # stage + diff, no push
heartbeatctl backup-identity --gc                  # git gc before push
```

Encryption uses an SSH-key recipient fetched from `github.com/<owner>.keys`
at scaffold time. Without a configured recipient, the primitive falls
back to **partial mode** — plaintext files committed, `.env.age` omitted,
visible in `heartbeatctl status`. Full spec at
`docs/superpowers/specs/2026-04-22-identity-backup-design.md`.

### `backup-vault`

Snapshot the markdown subset of the vault to `backup/vault`.

```
heartbeatctl backup-vault              # default: run
heartbeatctl backup-vault --dry-run    # stage + hash, no push
heartbeatctl backup-vault --gc         # git gc before push
```

The vault location comes from `vault.path` in `agent.yml` (default
`.state/.vault`, mapped to `/home/agent/.vault` inside the container by
the bind-mount). Files are filtered by `*.md` extension; the
`.obsidian/`, `.trash/`, and `*.sync-conflict-*` exclusions exist so
Syncthing-induced churn doesn't pollute the snapshots when the vault
is also synced to other devices.

Deletes propagate: the staged tree is wiped before each commit, so a
removed note drops out of the next `backup/vault` commit instead of
lingering forever.

### `backup-config`

Snapshot `agent.yml` to `backup/config`.

```
heartbeatctl backup-config              # default: run
heartbeatctl backup-config --dry-run    # stage + hash, no push
heartbeatctl backup-config --gc         # git gc before push
```

This branch is what `setup.sh --restore-from-fork` reads first during a
restore — `vault.path` and other rendered-file inputs all live in
`agent.yml`. Disable scheduled config backups via
`features.config_backup.enabled=false` in `agent.yml`.

## Exit codes (global)

- `0` — success
- `1` — validation error (bad input value, unknown schema)
- `2` — operational error (filesystem, yq, missing state.json)
