# heartbeatctl — reference

CLI for inspecting, controlling, and mutating the in-container heartbeat. Image-baked at `/opt/agent-admin/scripts/heartbeatctl`, symlinked to `/usr/local/bin/heartbeatctl`.

Full subcommand list (as of v0.12.0, matches `docker/scripts/heartbeatctl` dispatch): `status`, `logs`, `show`, `test`, `pause`, `resume`, `reload`, `kick-channel`, `drop-plugin`, `set-interval`, `set-prompt`, `set-notifier`, `set-timeout`, `set-retries`, `backup-identity`, `backup-vault`, `backup-config`, `token-check`, `qmd-reindex`, `wiki-graph`, `help`.

## Invocation

**Docker mode — always pass `-u agent` to docker exec.** The container runs with `cap_drop: ALL` + `no-new-privileges: true`, and several heartbeatctl paths (`reload`, `pause`, `set-*`) write to agent-owned files. The default `docker exec` is root, which cannot write those without `CAP_DAC_OVERRIDE` / `CAP_FOWNER`.

```bash
docker exec -u agent <agent-name> heartbeatctl <subcommand> [args]
# Inside the tmux session it's on PATH:
heartbeatctl <subcommand> [args]
# From the host, agentctl proxies any subcommand:
./scripts/agentctl heartbeat <subcommand> [args]
```

**Local mode (deployment.mode: local, features 011-013): there is no heartbeatctl binary on the host.** Only three actions have local equivalents, dispatched by `agentctl heartbeat <sub>` as direct execs of the rendered `scripts/local/agent-*.sh` wrappers (no `systemctl`, no polkit):

- `agentctl heartbeat qmd-reindex` — refuses `--dry-run` (the local entrypoint has none; it would run a real reindex).
- `agentctl heartbeat backup-vault` — passes `--dry-run` through.
- `agentctl heartbeat wiki-graph`

Every other `agentctl heartbeat <sub>` (including `token-check`) is docker-only and exits `2` in local mode. Local schedules run as systemd `OnCalendar` timers rendered from `modules/local-*.timer.tpl`, not cron — see the per-command notes below.

Local kill-switch: the rendered `scripts/local/agent-killswitch.sh` stops the main unit **and** every companion unit that carries the work documented here — `agent-<name>-qmd-reindex.timer`, `agent-<name>-qmd-watch.service`, `agent-<name>-vault-backup.timer`, `agent-<name>-wiki-graph.timer`, `agent-<name>-healthcheck.timer` (013 FR-008). Pass `--disable` to also prevent them starting at boot. Stopping only the main unit would leave the vault backup pushing to the fork long after you thought the agent was off.

## Read commands

### `status [--json]`

Dashboard. Reads `state.json` and enriches with live `crond.alive` / `pid` at read time.

Pretty output (no flag):

```
heartbeat: ENABLED
  interval: 2m   cron: */2 * * * *
  prompt:   Check status and report to notification telegram bot
  notifier: telegram
  last run: ok  @ 2026-04-19T19:36:00Z  (125000ms, attempt 1)
  counters: runs=2 ok=2 timeout=0 error=0 streak_fail=0  24h=100%
  crond:    alive (pid 16)
  updated:  2026-04-19T19:38:05Z

recent runs:
  2026-04-19T19:19:14Z  ok  73000ms   Check status and report...
  2026-04-19T19:36:00Z  ok  125000ms  Check status and report...
```

`--json`: emits `state.json` enriched with a live `.crond = {alive, pid}` overlay, re-serialized through `jq -c` — machine-parseable, but not byte-identical to the file (`state.json` itself always stores `crond: {alive: null, pid: null}`; the runner never knows crond's state).

Exit codes:

- `0` on success.
- `1` if `state.json` schema is unknown (refuses to guess at incompatible formats).
- `2` if `state.json` is missing (heartbeat has never run — try `heartbeatctl reload`).

### `logs [-n N] [--json]`

Tail `logs/runs.jsonl`. Default `N=20`.

Pretty output (no flag): a table of `TS | STATUS | DUR | ATTMPT | PROMPT[:60]`.

`--json`: emits raw JSON lines, one per run, piping-safe for `jq`.

### `show`

Dumps the three primary sources of truth so you can tell what's configured vs. what's staged:

- `heartbeat.conf` (shell-sourced by `heartbeat.sh`, regenerated by `reload`)
- the staging crontab (`scripts/heartbeat/.crontab.staging` — the *desired* schedule written by `reload`)
- `agent.yml` slice: `features.heartbeat` + `notifications`

Note `show` does **not** read `/etc/crontabs/agent` (what crond actually runs, root-owned). During the propagation window (~75s worst case, see "Propagation timing"), or if the sync loop is broken, the installed crontab can lag what `show` displays.

## Control commands

### `test [--prompt "..."]`

Run one heartbeat right now. Writes a line to `runs.jsonl` with `trigger: manual`. Counters are trigger-agnostic: a manual test updates `state.json` exactly like a cron tick — a failed test increments `counters.consecutive_failures` (and `total_runs`), a successful one resets the failure streak. `trigger: manual` is recorded in the run line only; the counter update never consults it.

Accepts `--prompt "override"` to test a different prompt without changing `agent.yml`.

Returns when the run completes — expect the typical duration (same as a cron tick).

### `pause`

Sets `features.heartbeat.enabled = false` in `agent.yml`, prefixes the crontab line with `# paused:`, and drops the cron dispatch. Idempotent — running it twice does not double-comment.

### `resume`

Inverse of `pause`. Sets `enabled = true`, uncomments the crontab line.

### `reload`

Re-reads `agent.yml`, regenerates `heartbeat.conf`, rewrites the crontab staging file, and ensures `logs/` exists. Does **not** signal crond (busybox crond dies on SIGHUP). crond picks up the new crontab on its own minute tick — propagation is described in the "Propagation timing" section below.

Idempotent. Called automatically by:

- `entrypoint.sh` on every container start (via `start_services.sh`).
- Every `pause` / `resume` / `set-*` subcommand after mutating `agent.yml`.

### `kick-channel`

Kills the `agent` tmux session; the `start_services.sh` watchdog respawns it within ~2s with a fresh plugin attachment. This is the manual recovery for the silent-stuck Telegram bridge case (bun alive and polling, but MCP channel notifications not reaching the session — automated detection was reverted for false positives). Exit `2` if tmux or the session is absent. Docker mode only (the tmux session lives in the container).

### `drop-plugin <spec>`

Removes a plugin spec (e.g. `caveman@JuliusBrussee`) from `agent.yml.plugins[]`. Idempotent (no-op with exit `0` if the spec isn't listed), atomic with `agent.yml.prev` backup/restore on `yq` failure. Does **not** auto-kick the channel — run `heartbeatctl kick-channel` afterwards so the next watchdog respawn consumes the new plugin list.

## Mutation commands

All mutations are transactional with automatic rollback on failure:

1. Validate the value. Reject with exit code 1 if invalid; `agent.yml` untouched.
2. Back up `agent.yml` to `agent.yml.prev`.
3. Apply the mutation via `yq -i`.
4. Run `reload`.
5. On any failure in steps 3–4, restore `agent.yml.prev` and re-run `reload` so the live state matches the backed-up config. Exit code 2.
6. On success, delete the backup.

### `set-interval <Nm|Nh>`

Accepts a restricted divisor table so the derived cron expression is always valid:

- `Nm` for `N` in `{1, 2, 3, 4, 5, 6, 10, 12, 15, 20, 30}` → `*/N * * * *` (or `* * * * *` for `1m`).
- `Nh` for `N` in `{1, 2, 3, 4, 6, 8, 12}` → `0 */N * * *` (or `0 * * * *` for `1h`).
- `24h` → `0 0 * * *` (once per day at 00:00 UTC).

Anything else is rejected. Examples that do NOT work: `30s`, `45m`, `7m`, `5h`, `2h30m`.

### `set-prompt "<str>"`

Updates `features.heartbeat.default_prompt`. Accepts the prompt as an arg or via stdin if no arg is given. Length 1..4000 chars.

Warns to stderr (but does not refuse) if the current `notifications.channel` is `telegram` and `.env` is missing `NOTIFY_BOT_TOKEN` / `NOTIFY_CHAT_ID`.

#### Prompt examples by use case

The default prompt (`Status check — return a short plain-text report (uptime, notable issues). No tool use; your stdout is forwarded verbatim to the notifier.`) is intentionally generic. Replace it via `heartbeatctl set-prompt "..."` to match what your agent actually does:

- **Cluster health** — *"List Kubernetes pods in CrashLoopBackOff or Error in the last 30m via `kubectl get pods -A`. Return only the failing pods with their namespace and reason. If everything is healthy, output `OK: N pods running`."*
- **Vault ingestion monitor** — *"Count `*.md` files under `~/.vault/wiki/` and compare against the last value in `~/.vault/log.md`. Report any drop or > 10% gain. Note any sync-conflict files in `~/.vault/`."*
- **Jira sprint blockers** — *"Use `mcp__atlassian__jira_search` to query the active sprint of project KEY. List blockers (status=Blocked or label=blocker). Group by assignee. Output one bullet per blocker with `[KEY-123] @user — title`."*
- **Log tailer** — *"Tail the last 50 lines of `/workspace/claude.log` and report any ERROR or WARN. Include timestamps and a one-line summary of patterns (e.g. `3× 'connection refused' between 14:05 and 14:18`)."*
- **Heartbeat-as-canary** — *"Reply with `pong @ <ISO timestamp>`. No tool use. This proves Claude Code is reachable and the notifier round-trip works."*

Each prompt should:
1. Be self-contained — heartbeat sessions run in an isolated `CLAUDE_CONFIG_DIR=/home/agent/.claude-heartbeat` with no plugins and no conversation history. Auth (`.credentials.json`) and `.claude.json` are symlinked from the interactive session's config dir; `settings.json` is rewritten with `enabledPlugins = {}` and `plugins/` is an empty directory (not a symlink), so `claude --print` never spawns a channel plugin that could SIGTERM the interactive session's bun.
2. Constrain output length — heartbeat trims at 3500 chars before forwarding to the notifier; prompts that ramble waste the budget.
3. Be explicit about tools — say *"no tool use"* when you want a fast pulse, or *"use `mcp__X__Y`"* when you want a query.

### `set-notifier <none|log|telegram>`

Updates `notifications.channel`. Membership check — anything else rejected.

- `none` — silent; no-op notifier.
- `log` — appends to `scripts/heartbeat/logs/notifications.log`.
- `telegram` — sends via `NOTIFY_BOT_TOKEN` / `NOTIFY_CHAT_ID` (see note in `set-prompt`).

### `set-timeout <seconds>`

Integer ≥ 10. Max timeout the heartbeat waits for `claude --print` before killing the tmux session and recording `status: timeout`.

### `set-retries <N>`

Integer 0..5. Number of retries after a failed attempt. Total attempts = `retries + 1`.

## Propagation timing

When you mutate a field, the live schedule updates in two steps:

1. `heartbeatctl reload` writes the new crontab to the workspace staging file (`scripts/heartbeat/.crontab.staging`). Instantaneous.
2. The root-privileged sync loop in `entrypoint.sh` notices the staging file differs byte-for-byte from `/etc/crontabs/agent` (`cmp -s`, not mtime — busybox `sh -nt` rounds to whole seconds and would miss same-second writes at boot) and copies it there. Runs every 15 seconds.
3. busybox crond re-scans crontab mtimes on its minute tick and picks up the new schedule. Up to 60 seconds.

**Worst case total propagation: ~75 seconds.**

The staging/sync indirection exists because busybox crond silently skips any file in `/etc/crontabs/` that isn't root-owned, so agent cannot write there directly. Entrypoint runs as root before dropping privileges, so it's the only context that can install files into `/etc/crontabs/`.

## Data files

All under `/workspace/scripts/heartbeat/` (as of v0.12.0):

Heartbeat core:

- `heartbeat.conf` — shell-sourced by `heartbeat.sh`. Regenerated by `reload`.
- `state.json` — atomic snapshot of the last run + aggregate counters. Schema `1`.
- `.crontab.staging` — written by `reload`, consumed by entrypoint sync loop.
- `logs/runs.jsonl` — append-only JSON-per-line trace. Rotated at 10MB → `.1`, `.2.gz`, `.3.gz` (max 3 generations).
- `logs/cron.log` — crond stderr for schedule-dispatch debugging.
- `logs/notifications.log` — output of the `log` notifier.
- `logs/sessions/*.log` — per-run tmux session output (last 20 retained).

Per-primitive state files (same directory):

- `identity-backup.json`, `vault-backup.json`, `config-backup.json` — hash + `last_commit` + `last_push` per backup branch.
- `token-health/<id>.json` — one per probed token, plus `token-health/warnings.jsonl` (warning audit trail).
- `qmd-index.json` — reindex state, schema `{hash, last_run, last_status, runs[, pending]}` (see "Vault RAG" below).
- `wiki-graph.json` + `.wiki-graph.lock` — wiki-graph freshness/counts state and its flock file.

Per-primitive cron logs (docker mode; the cron lines written by `reload` redirect here):

- `logs/backup-identity.log`, `logs/backup-vault.log`, `logs/backup-config.log`, `logs/token-health.log`, `logs/qmd-reindex.log`, `logs/wiki-graph.log`.

The qmd reindex lock is the exception to the "all under `scripts/heartbeat/`" rule: it lives at `<qmd cache root>/.reindex.lock`, shared with the first-boot setup so setup and reindex never overlap. The cache root is `$QMD_CACHE_HOME`, falling back to `~/.cache/qmd` — docker mode: `HOME=/home/agent`, which is the `.state/` bind-mount; local mode: the rendered entrypoints export `QMD_CACHE_HOME=<workspace>/.state/.cache/qmd`. Either way it persists under `.state/`.

## Backup commands

The agent's persistent state is sliced into three orphan branches on the
agent's own fork (`backup/identity`, `backup/vault`, `backup/config`).
Each primitive is idempotent, hash-based, and pushes nothing when its
input hasn't changed since the last commit. Restore happens via
`setup.sh --restore-from-fork <url>`, which pulls all three in order.

| Branch | Cadence (docker mode: cron lines written by `reload`) | What it stores |
|---|---|---|
| `backup/identity` | watchdog 60s + post-plugin-install + cron `30 3 * * *` (override via `features.identity_backup.schedule`) | OAuth login, Telegram pairing, plugin config, settings, age-encrypted `.env` |
| `backup/vault` | cron `0 * * * *` (override via `vault.backup_schedule`) | Markdown subset of the vault. Excludes `.obsidian/workspace*.json`, cache, `.trash/`, `*.sync-conflict-*` |
| `backup/config` | cron `30 3 * * *` (toggle via `features.config_backup.enabled`) | `agent.yml` (plaintext, no secrets) |

Local mode: only the **vault** backup is scheduled — as a systemd timer (`agent-<name>-vault-backup.timer`, `OnCalendar` converted from `vault.backup_schedule`). Identity and config backups have no local scheduler (no `local-*` unit templates exist for them); run them from a docker deployment or not at all.

> **Freshness check**: `agentctl doctor` verifies that each backup pushed within its expected cadence × 2 (identity 48h, vault 25h, config 8d — local mode checks vault only) and prints a humanized delta (`pushed 12h ago`); a stale or never-pushed backup is a warning (exit `1`), not a failure. Docker mode: override the thresholds for one-off runs via `DOCTOR_IDENTITY_MAX_AGE_HOURS`, `DOCTOR_VAULT_MAX_AGE_HOURS`, `DOCTOR_CONFIG_MAX_AGE_DAYS`. Local mode ignores those env vars — its vault check passes a fixed 25h. See "agentctl doctor exit contract" below. `heartbeatctl status` prints the same `last_push` timestamps for all three branches under their respective sections.

### `backup-identity`

Snapshot the agent's identity subset (login, pairing, plugin config,
settings, optionally age-encrypted `.env`) to the `backup/identity`
orphan branch on the fork.

```
heartbeatctl backup-identity                       # default: run
heartbeatctl backup-identity --configure-key KEY   # set recipient + backup
heartbeatctl backup-identity --disable             # stop scheduled backups
heartbeatctl backup-identity --dry-run             # stage + diff, no push
heartbeatctl backup-identity --gc                  # git gc before push
```

Encryption uses an SSH-key recipient fetched from `github.com/<owner>.keys`
at scaffold time. Without a configured recipient, the primitive falls
back to **partial mode** — plaintext files committed, `.env.age` omitted,
visible in `heartbeatctl status`. Full spec at
`docs/superpowers/specs/2026-04-22-identity-backup-design.md`.

### `backup-vault`

Snapshot the markdown subset of the vault to `backup/vault`.

```
heartbeatctl backup-vault              # default: run
heartbeatctl backup-vault --dry-run    # stage + hash, no push
heartbeatctl backup-vault --gc         # git gc before push
```

The vault location comes from `vault.path` in `agent.yml` (default
`.state/.vault`, mapped to `/home/agent/.vault` inside the container by
the bind-mount). Files are filtered by `*.md` extension; the
`.obsidian/`, `.trash/`, and `*.sync-conflict-*` exclusions exist so
Syncthing-induced churn doesn't pollute the snapshots when the vault
is also synced to other devices.

Deletes propagate: the staged tree is wiped before each commit, so a
removed note drops out of the next `backup/vault` commit instead of
lingering forever.

### `backup-config`

Snapshot `agent.yml` to `backup/config`.

```
heartbeatctl backup-config              # default: run
heartbeatctl backup-config --dry-run    # stage + hash, no push
heartbeatctl backup-config --gc         # git gc before push
```

This branch is what `setup.sh --restore-from-fork` reads first during a
restore — `vault.path` and other rendered-file inputs all live in
`agent.yml`. Disable scheduled config backups via
`features.config_backup.enabled=false` in `agent.yml`.

## Token health

Periodically probes the free-tier auth endpoints of every configured
external token, so you find out a PAT expired *before* Claude tries to
use it and dies with a 401 mid-task.

Probes (free, no quota cost):

| Token              | Probe                                                 | Detects                |
|--------------------|-------------------------------------------------------|------------------------|
| `GITHUB_PAT`       | `GET https://api.github.com/user`                     | expired/revoked PAT    |
| `NOTIFY_BOT_TOKEN` | `GET https://api.telegram.org/bot<TOKEN>/getMe`       | revoked Telegram bot   |
| `ATLASSIAN_*_TOKEN`| `GET <jira_url>/rest/api/3/myself` (basic auth)       | expired Atlassian token|
| Claude Code OAuth  | local read of `~/.claude/.credentials.json` `expiresAt` — no network call | expired OAuth, or expiring within 30 min (early warn, so you can `/login` before the next heartbeat 401s) |

The claude_oauth probe writes `token-health/claude_oauth.json` like the
others, and maps both "expired" and "expires in <30 min" to `auth_fail`
so the same warning/dedup pipeline applies. The runner only invokes it
when `~/.claude/.credentials.json` exists (override the path with
`TH_CLAUDE_CRED_OVERRIDE`) — a never-logged-in agent gets no
`claude_oauth.json` at all rather than a `skipped` one. A file that
exists but has no parseable `expiresAt` does record `skipped`.

Firecrawl and Google Calendar are NOT probed automatically — Firecrawl
charges per call and Google's OAuth refresh path is fragile. Run a
manual probe with `agentctl heartbeat token-check` if you suspect either,
or rely on Claude's own error reporting when it hits the MCP.

### `token-check`

```
heartbeatctl token-check
agentctl heartbeat token-check          # same, from the host (docker mode only)
```

Local mode has no token-health scheduler and `agentctl heartbeat
token-check` exits `2` there — token health is a docker-mode feature.

Triggers a one-shot probe of all configured tokens. In docker mode an
hourly cron line does the same automatically (default schedule
`0 * * * *`). State files land
in `<workspace>/scripts/heartbeat/token-health/<id>.json`:

```json
{
  "id": "github",
  "kind": "github_pat",
  "last_check": "2026-05-02T22:00:00Z",
  "status": "ok",
  "http_code": "200",
  "latency_ms": 142,
  "consecutive_failures": 0,
  "first_failure_at": null,
  "last_warned_at": null,
  "error": null
}
```

`status` ∈ `{ok, auth_fail, network, skipped}`. Only `auth_fail` and
sustained `network` failures emit warnings — the docker-mode `agentctl doctor`
adds checks 16-18, which read the same files (github, telegram, every
`atlassian-*`, and `claude_oauth` when present) and surface them as `✗`/`⚠`. A
probe file older than `DOCTOR_TOKEN_MAX_AGE_HOURS` (default `2`) is itself a
warning — a dead cron is as bad as a dead token.

### Warnings

When a token transitions from `ok` → `auth_fail`, the runner emits ONE
warning through the configured heartbeat notifier (Telegram if set,
log otherwise). The same warning re-fires every 24h if the failure
persists, so the user gets a daily reminder without spam. Recovery
(`auth_fail` → `ok`) emits a single "recovered" notice.

Warning history is appended to
`<workspace>/scripts/heartbeat/token-health/warnings.jsonl` regardless
of the notifier outcome, so the audit trail survives a broken Telegram
channel.

### Disable / tune

```yaml
# agent.yml
features:
  token_health:
    enabled: false        # opt out entirely
    schedule: "*/15 * * * *"   # custom cron, default "0 * * * *"
```

Override the dedup window per-invocation: `TH_DEDUP_SECS=3600
heartbeatctl token-check` (default 86400 = 24h).

## Exit codes (global)

- `0` — success
- `1` — validation error (bad input value, unknown schema)
- `2` — operational error (filesystem, yq, missing state.json)

## Vault RAG

### `qmd-reindex [--dry-run]`

```
heartbeatctl qmd-reindex                # docker mode, in-container
agentctl heartbeat qmd-reindex         # both modes, from the host
```

Reindexes the vault's QMD semantic search (feature 010; local parity 013).
Opt-in: requires **both** `vault.enabled: true` and `vault.qmd.enabled: true`
in `agent.yml`. Fail-silent — the run path always exits `0` (a cron/watcher
tick must never crash); honesty is in the state file.

Two schedulers invoke the same command:

- Docker mode: a cron backstop, default `*/5 * * * *` (override via
  `vault.qmd.schedule`), plus the inotify watcher (`qmd_watch.sh`, started by
  `start_services.sh`) on vault changes.
- Local mode: the `agent-<name>-qmd-reindex.timer` systemd unit (`OnCalendar`
  converted from `vault.qmd.schedule`; a non-convertible schedule falls back to
  the feature default and drops a `qmd-schedule.fallback` marker that `doctor`
  warns about) plus the `agent-<name>-qmd-watch.service` watcher.

Overlap is safe: the critical section is `flock`-guarded
(`<qmd cache root>/.reindex.lock`, shared with the first-boot setup) and
hash-debounced (`vault_hash` over the markdown subset — the same hash the vault
backup uses).

`--dry-run` (docker mode): resolve + hash + report whether a run would reindex;
no embed, no state write. The local wrapper has no dry-run mode, so
`agentctl heartbeat qmd-reindex --dry-run` is refused in local mode (exit `2`)
rather than silently running a real reindex.

#### Embed completion loop (018)

A single `qmd embed` runs inside an engine session hard-capped at ~30 minutes;
on a large first-time corpus one pass dies with "LLM session expired", leaving
part of the corpus without vectors. As of v0.12.0 the reindex runs successive
fresh `qmd embed` passes inside one locked invocation until one of:

- **complete** — a pass prints `All content hashes already have embeddings`,
  or `qmd status` reports `Pending: 0` → `last_status: indexed`, `pending: 0`.
- **stalled** — pending did not decrease between consecutive passes
  (permanently-failing documents) → `last_status: stalled`, pending persisted.
- **pass cap** — `QMD_EMBED_MAX_PASSES` reached (fixed constant, `12` as of
  v0.12.0; env-overridable for tests only — deliberately **not** an `agent.yml`
  field) → `last_status: partial`, `pending` = the last count parsed from
  `qmd status` (`0` if no pass ever produced a parseable count).
- **pass failure/timeout** → `last_status: error`, and the *previous* hash is
  kept so the next tick re-runs `update` from scratch. `pending` is carried
  over from the prior state file unchanged (including its absence).

#### `qmd-index.json` state schema

```json
{
  "hash": "<sha256 over vault markdown content + filenames>",
  "last_run": "2026-07-12T18:00:00Z",
  "last_status": "indexed",
  "runs": 42,
  "pending": 0
}
```

- `last_status` ∈ `{indexed, skipped, error, partial, stalled}`.
- `pending` — documents still needing an embedding, parsed from `qmd status`
  (`Pending: N`). Absent on pre-018 state files, which readers treat as
  *unknown*, never as zero.
- `runs` increments on every state write, regardless of status.

Resume guarantee (018): the unchanged-vault guard skips embedding **only** when
the hash is unchanged *and* `pending == 0` (then writes `skipped`). An
unchanged vault with `pending > 0` — or with `pending` unknown — resumes the
embed loop without re-running `update`. A `partial` or `stalled` first index
therefore self-heals on subsequent ticks even when the vault never changes.

Local mode: `agentctl doctor` degrades to a warning when `last_status` is
`error`; `partial` and `stalled` are reported as a **pass** line with the status
in parentheses (as of v0.12.0 — they are self-healing states, not failures), and
`agentctl status` shows the index presence, the last-reindex timestamp and its
status. See "agentctl doctor exit contract" below.

### `wiki-graph`

```
heartbeatctl wiki-graph
agentctl heartbeat wiki-graph           # same, from the host (both modes)
```

Regenerates the derived wiki graph + structural lint (feature 014). Reads the
whole `wiki/` and writes `<vault>/.graph/{graph,backlinks,findings}.json` plus the
`<workspace>/scripts/heartbeat/wiki-graph.json` state file (freshness + finding
counts). `flock`-guarded (lock lives under `scripts/heartbeat/`, never in the
vault), fail-silent (always exits 0; honesty is in the state file). NEVER edits the
wiki — it reports; the agent fixes. A `20 */6 * * *` backstop runs the same command
(docker mode: cron line; local mode: the `agent-<name>-wiki-graph.timer` unit;
override via `vault.wiki_graph.schedule`); opt out with
`vault.wiki_graph.enabled=false`.

Local mode: `agentctl status`/`doctor` surface the freshness and degrade on
integrity findings (broken links, frontmatter violations, index drift → warn)
or a dead runner (fail). Docker mode (as of v0.12.0): neither surfaces the wiki
graph — `agentctl status` proxies `heartbeatctl status`, which reports the
heartbeat, backups and token health only, and `agentctl doctor` has no
wiki-graph checks. Read `scripts/heartbeat/wiki-graph.json` directly. See
"agentctl doctor exit contract" below.

## agentctl doctor exit contract

Not a heartbeatctl subcommand, but the consumer of most state files documented
above. `agentctl doctor` aggregates pass/warn/fail checks and exits (both
modes, as of v0.12.0):

- `0` — all checks passed
- `1` — warnings only
- `2` — at least one failure

Checks over this CLI's surface:

| Check | Degrades to | Mode |
|---|---|---|
| Backup never pushed / stale (identity 48h, vault 25h, config 8d) | warn | docker (vault only in local) |
| Token probe `auth_fail` (incl. claude_oauth expired or expiring <30 min) | fail | docker |
| Token probe `network` failure, missing or stale probe file | warn | docker |
| QMD index not built yet | warn | local |
| QMD last reindex `last_status: error` | warn | local |
| QMD watcher unit failed (timer still backstops) | warn | local |
| QMD / wiki-graph schedule fallback marker present | warn | local |
| wiki-graph integrity findings (broken links, frontmatter violations, index drift) | warn | local |
| wiki-graph runner dead (last run older than 2x the schedule interval) | fail | local |

The QMD and wiki-graph rows are checked by the local-mode doctor only: as of
v0.12.0 the docker-mode doctor has no qmd/wiki-graph checks (its RAG-adjacent
coverage is the vault-skeleton check plus the three backup-freshness checks).
`last_status: partial` / `stalled` do not degrade the exit code in either mode —
only `error` does.
